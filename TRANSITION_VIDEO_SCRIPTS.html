1. Main Menu — __MainMenu__.unity

What to say in the video

  • The main menu is the navigation hub. It has buttons that load each of the 6 simulation scenes.
  • The scene uses UIController.cs (in Assets/Scripts/Main/). It stores a list of ButtonScenePair objects — each pair maps a UI button to a scene name string.
  • To add a new scene: (1) build the scene, (2) add it to Build Settings, (3) drop the button into UIController's list and type the exact scene name.
  • LoadScene.cs is an alternate version of the same logic — check which one is actually attached to the GameObject in the scene before editing.
  • SceneReference.cs is a utility that stores the asset path at editor time so scene names don't break if scenes are renamed — use it if you want safer scene references in the future.

2. Electromagnetic Waves — __EMWaves__.unity

What the scene teaches

Visualizes a propagating electromagnetic wave in 3D — a red Electric field wave and a blue Magnetic field wave, perpendicular to each other, traveling along the Z-axis. Users can manipulate amplitude, wavelength, and frequency in real time.

Physics implemented

E(z,t) = A · sin(2π/λ · z − ω·t) [Electric field, y-direction]
B(z,t) = A · sin(2π/λ · z − ω·t) [Magnetic field, x-direction]
Wave speed = 2π · λ · f
k = 2π / λ (stored in GlobalVariables.k)

What to say in the video

  • The main visualization is driven by EMWaveManager.cs. It creates a 3D grid of field points and places arrow objects at each one using the custom Arrow.cs class.
  • There are actually two visualization systems: EMWAVE.cs draws the classic single sine-wave line (the curved ribbon you see), while EMWaveManager places the vector arrows in the field grid. Know which GameObjects in the scene use which script.
  • Object pooling is used in EMWaveManager — field point GameObjects are pre-allocated and reused. Do not destroy them at runtime. If arrows disappear, check the pool size (waveRows, pointCountPerWave).
  • Distance culling: Field arrows far from the camera are automatically hidden for performance. This is controlled inside EMWaveManager — if you zoom way out and arrows disappear, that's intentional.
  • Peak color highlighting: Arrow heads change color (darken) when the field is at a peak. This is handled in EMWaveManager's Update() loop — it calls arrow.SetHeadColor() on Arrow.cs.
  • Constant wave speed option: There's a toggle that locks c = λ·f constant. When active, changing wavelength auto-adjusts frequency and vice versa. This is important for physics accuracy — mention it to future devs.
  • GlobalVariables.cs is a static class that ALL scenes share. It holds lambda, omega, E0, B0, theta, k. It also controls the global pause state via GlobalVariables.IsPaused. If you change the pause logic, every scene is affected.
  • The Pause/Play button is handled by PausePlay_EM_Waves.cs — it flips GlobalVariables.IsPaused and updates button text.
  • Reset button: ResetScript.cs shows a confirmation panel before reloading the scene. Every scene has this same reset flow.

3. Faraday's Law — __FaradayLaw__.unity

What the scene teaches

Shows a coil rotating in a magnetic field. As the coil rotates, the magnetic flux through it changes, which induces an EMF (voltage) and drives a current. Animated charge particles flow around the circuit loop to visualize the current direction.

Physics implemented

Magnetic Flux: Φ = B · A · cos(θ)
Faraday's Law: ε = −dΦ/dt (EMF = rate of flux change)
Ohm's Law: I = ε / R (Resistance fixed at 10 Ω)
Area: A = Length × Width

What to say in the video

  • Faraday.cs is the main controller. It's an extension of the Flux scene concept — it adds time-dependence (the coil rotates) and computes the EMF as the numerical derivative of flux over time (dΦ/dt via frame-by-frame difference).
  • The rotating 3D object is a cube that represents the coil cross-section — it is NOT a real mesh coil, just a visual proxy. This is worth mentioning so the next dev doesn't go looking for a coil prefab.
  • The areaArrow (blue/green arrow) represents the normal to the coil surface — its direction shows you which way the coil is facing. The fieldArrow (red) shows the magnetic field direction. The angle between them determines flux.
  • Frequency slider controls how fast the coil rotates. At frequency = 0, the coil is stationary (flux is constant, EMF = 0). At high frequency, the EMF oscillates rapidly — great for showing the AC generator analogy.
  • CurrentVisualizer.cs (in Assets/Assets/ChargePath/) animates the charge particles around the circuit loop. The speed of particles is proportional to the induced current magnitude. When current reverses sign, particles flow the other direction.
  • CircuitLoop.cs defines the path the particles travel. It can generate an ellipse or polygon in any of the three cardinal planes. The waypoints are fed to CurrentVisualizer. If the circuit loop looks wrong, check CircuitLoop's orientation and shape settings in the Inspector.
  • FaradayLawVariables.cs is a static store for the calculated Flux, EMF, and Current values. If you ever add a second UI element that needs these values elsewhere in the scene, read from here instead of recalculating.
  • TestCurrentControl.cs is a debugging tool — it has a manual slider (−50 to +50 A) that overrides the current value going into CurrentVisualizer. It should be disabled in the production scene. Check that it's not accidentally active before a demo.

Watch out for

  • TestCurrentControl.cs must be DISABLED in the production scene — if it's active it overrides real current values and the visualization won't match the physics.
  • EMF is computed as a frame-by-frame numerical derivative (current frame flux minus previous frame flux, divided by Time.deltaTime). This means at very low frame rates, the EMF display can look jittery — this is a known limitation, not a bug.
  • Resistance is hardcoded at 10 Ω in Faraday.cs — it's not exposed as a slider. If you need to make it adjustable, add a slider and wire it to the resistance variable.

4. Ampere's Law — __AmperesLaw__.unity

What the scene teaches

Visualizes the Amperian loop — the closed path you integrate the magnetic field around in Ampere's Law. Users can choose different conductor geometries and different loop shapes, and see how the loop wraps around the conductor in 3D.

Physics context

Ampere's Law: ∮ B · dl = μ₀ · I_enclosed
This scene visualizes the integration path (loop), not the field itself.

What to say in the video

  • Amperes.cs is the sole main script. It is large — it handles conductor selection, loop selection, all UI, and all geometry generation in one file. Be prepared for a dense read.
  • Conductor types: NoWire, Line (infinite straight wire), Sheet (planar current), Solenoid (helical coil), Toroid (donut shape). Each activates/deactivates different 3D model GameObjects in the scene — they are NOT generated procedurally, they are pre-placed prefabs/models toggled by this script.
  • Loop types: Circle and Rectangle (Segment type is coded but ignored/skipped). These ARE generated procedurally by the script using LineRenderer. BuildCircle() and BuildRectangle() are the methods to look at if the loop geometry needs changing.
  • The loop can be placed on three planes (XZ/floor, XY/front, YZ/side) and rotated freely 0–360°. There's also a Z-offset slider to slide the loop along the axis so you can show it passing through or outside the conductor.
  • The UI has dimension panels that toggle visibility: when Circle is selected, only the radius/resolution panel shows; when Rectangle is selected, width/height/segment panels show. This is controlled by panel GameObject activation in Amperes.cs — if you add a new loop type, you'll need to add a new panel and wire it up.
  • GetPointOnPlane() is a key helper method that takes a 2D (u, v) position and maps it into 3D space based on the selected orientation and rotation. All loop geometry goes through this. If the loop appears on the wrong plane or rotated wrong, this method is where to debug.
  • This scene does NOT numerically calculate ∮B·dl — it only shows the visualization path. If a future task is to add actual flux/circulation calculation, that's a new feature requiring Biot-Savart integration.

5. Gaussian Surface — __GaussianSurface__.unity

What the scene teaches

Gauss's Law: Shows how a Gaussian surface encloses a charge distribution, and how the enclosed charge determines the electric flux through the surface. Includes real-time field line visualization from the charge.

Physics implemented

Gauss's Law: Φ = Q_enclosed / ε₀ (ε₀ = 8.854×10⁻¹² F/m)
Enclosed charge: Q_enc = ρ × V_overlap (charge density × overlapping volume)
Overlap volume: Calculated geometrically between charge and surface colliders

What to say in the video

  • Control.cs is the master controller for this scene. It handles which charge type is active and which Gaussian surface is active, and updates the scene in real time. It's important — don't confuse it with Unity's built-in Control class.
  • Charge types: Point charge, Linear (line/cylinder), Planar (sheet), None. Each is a pre-placed GameObject that Control.cs activates/deactivates.
  • Gaussian surface types: Sphere, Cylinder1, Cylinder2, Cube, None. Same activation/deactivation pattern.
  • Transparency feedback: When the charge is larger than the Gaussian surface (so the surface is inside the charge), the transparency of the objects is swapped — the charge becomes more opaque and the surface becomes more transparent. This is a visual hint that the charge extends beyond the surface. This logic lives in Control.cs — look for the size comparison and material alpha changes.
  • GaussianFlux.cs does the actual math. It uses the SphereCollider component on the objects to calculate the overlap volume between charge and surface. The OverlappingVolume() method is the geometric intersection calculation. The result feeds directly into the flux display.
  • EFieldLines.cs generates the electric field lines. It uses three different generation strategies: (1) Fibonacci sphere distribution for point charges, (2) radial lines for cylinders, (3) perpendicular lines for planar charges. Field lines use object pooling — LineRenderer objects are pre-allocated. Lines are also frustum-culled (disabled when off-screen) for performance.
  • UniformSphereDistribution.cs uses the golden angle / Fibonacci sphere formula to place exactly 100+ points uniformly on a sphere surface. This is used as the starting positions for the sphere's field lines. The formula is φ = π(3 − √5).

6. Gaussian Flux / Electric Flux — __GaussianFlux__.unity

What the scene teaches

A simpler, more direct demonstration of electric flux: Φ = E · A · cos(θ). A flat loop sits in a uniform electric field and users can control the field strength, loop area, and the angle between the field and the loop's normal vector.

Physics implemented

Electric Flux: Φ = E · A · cos(θ − 90°)
Area: A = Length × Width
Note: angle is measured from the loop normal to the field direction.

What to say in the video

  • Flux.cs is the main script. Think of this as the simpler predecessor to the Faraday scene — same angle/area/field concept, but static (no time evolution or coil rotation unless you activate the auto-rotation).
  • There are two arrows: areaArrow (shows the surface normal vector, scaled by area magnitude) and fieldArrow (shows the E-field direction and magnitude). These are Arrow.cs instances — the same utility used across the whole project.
  • The angle displayed is the angle between the loop's normal and the X-axis, not the angle between the normal and the field. Keep this distinction in mind when explaining to students — the formula uses the angle between normal and field, which is derived internally.
  • Snap points: The rotation has optional snap-to-angle behavior at key angles (0°, 90°, 180°, 270°). This helps students see the extreme cases (Φ = max at 0°, Φ = 0 at 90°). Check if snap is enabled in the Inspector.
  • PausePlay.cs in this scene pauses the auto-rotation of the loop. Unlike the EM Waves pause, this one also disables the rotation sliders during pause to prevent conflicting input.
  • AnglesBetweenVectors.cs is used to render the visual angle sector (the pie-slice shape you see between the field and normal arrows). It procedurally generates a sector mesh and can display the angle value. It's a self-contained utility.
  • FieldLines.cs creates the background uniform E-field visualization — a 4-directional grid of parallel arrow lines (top, bottom, left, right of the viewport). This is purely cosmetic to represent the uniform external field.

7. Hall Effect — __HallEffect__.unity

What the scene teaches

When a current-carrying conductor is placed in a perpendicular magnetic field, the charge carriers (electrons or holes) are deflected by the Lorentz force and accumulate on one side of the conductor. This creates a transverse voltage called the Hall voltage. Users can see this accumulation happen in real time with animated particles.

Physics implemented

Lorentz Force: F = q(E + v × B)
Hall E-field: E_Hall = −σ / ε (σ = lateral charge surface density)
Net lateral force = q·E_Hall + q·v × B → equilibrium when they cancel
Carrier type: electrons (q = −e) deflect opposite to holes (q = +e)

What to say in the video

  • HallEffect.cs is the main and only controller script. It is the most complex script in the project — it handles particle emission, physics integration, pooling, and UI all in one file. Plan extra time for this section of the handoff.
  • The scene setup is two parallel plate objects (cathode and anode). Particles spawn on the cathode (or anode depending on carrier type) and drift toward the opposite plate, representing the flow of current through the conductor slab.
  • Object pooling: 500 particles per carrier type are pre-allocated at Start(). Do NOT reduce this number without profiling first — the emission rate scales with voltage, so at high voltages you can exhaust the pool.
  • Particle emission is grid-based: 256 spawn positions are calculated on the cathode surface and shuffled randomly each emission cycle. This prevents visual clustering at the same point.
  • Carrier type (Electron vs Hole): Electrons have q = −e, holes have q = +e. Changing the carrier type reverses the direction of the Lorentz deflection — electrons accumulate on one side, holes on the other. This is the key teaching point of the Hall Effect.
  • The Hall E-field accumulation is simulated: as particles hit the side walls, they are counted and their charge contributes to a growing σ (surface charge density). This σ creates a counter-field E_Hall = σ/ε that eventually balances the Lorentz force, reaching equilibrium. You can watch the particle deflection decrease over time as this equilibrium builds up.
  • Boundary condition: When a particle hits a lateral wall, it clamps to the wall and slides along it. When it hits the front/back face (anode), it returns to the pool and is marked inactive. This is important — "sliding on walls" behavior is intentional physics, not a bug.
  • Relative permittivity is set to 11.7 (silicon) by default. This affects the strength of the Hall E-field counter-force. If you're modeling a different material, this is the parameter to change.
  • ChargeParticle.cs is just a data class (no MonoBehaviour) — it holds each particle's CarrierType, Velocity, and Age. All actual movement logic is in HallEffect.cs.
  • ChargeEmitter.cs is fully commented out — it was an older separate emission system. The current emission logic is entirely inside HallEffect.cs.

Shared Utilities — Know These for ANY Scene

Arrow.cs — Universal Vector Visualization

  • Location: Assets/Assets/Arrow/Arrow.cs
  • Used by EVERY scene that shows vector arrows (E-field, B-field, area normal, etc.).
  • Creates a head (cone) + tail (cylinder) programmatically. If prefabs are missing, it falls back to procedurally-generated meshes.
  • Supports optional 3D text labels that follow the arrowhead and maintain correct scale even when the parent is stretched.
  • Key method: SetTailLength(float) to resize the arrow dynamically. SetHeadColor(Color) for peak highlighting.
  • Arrow.cs has try-catch blocks — if you see silent failures (arrow not appearing), check the Unity console for caught exceptions from this class.

CircuitLoop.cs + CurrentVisualizer.cs — Current Animation

  • Location: Assets/Assets/ChargePath/Scripts/
  • Used by Faraday's Law scene (and potentially Ampere's Law in the future).
  • CircuitLoop generates waypoints; CurrentVisualizer moves particles along them at a speed proportional to current magnitude.
  • Speed formula: speed = (currentAmps × sensitivity) / pathLength per frame.
  • Particles wrap around the loop continuously — when a particle reaches the end of the waypoint list it resets to the start.

ResetScript.cs — Universal Reset

  • Location: Assets/Scripts/Waves-Scripts/ResetScript.cs
  • Shared across all scenes. Shows a confirmation panel → reloads current scene.
  • On reset, all static variables (GlobalVariables) retain their last value until explicitly reset. If you need clean-state resets, add initialization code to each scene's Start() method.

GlobalVariables.cs — Cross-Scene State

  • Location: Assets/Scripts/Waves-Scripts/GlobalVariables.cs
  • Static class — lives as long as the application runs.
  • Holds: lambda, omega, E0, B0, theta, k.
  • Controls the global pause via GlobalVariables.IsPaused.
  • Because it's static, values from EM Waves scene carry over if the user navigates to another scene without restarting. Each scene's Awake/Start should set sensible defaults for the variables it needs.

Project-Wide Architecture Notes

Dual Slider + Input Field Pattern

Every controllable parameter in every scene has BOTH a slider AND a text input field. They are kept in sync: typing in the field updates the slider, and dragging the slider updates the field. This sync logic is duplicated in each scene's main script — there is no shared helper class for it. If you add a new parameter, follow this same pattern.

Object Pooling Pattern

EMWaveManager, EFieldLines, and HallEffect all use object pooling. The pattern: pre-allocate a fixed list of GameObjects in Start(), disable unused ones, and re-enable/reposition them as needed. Never call Instantiate() or Destroy() at runtime in these scripts — always use pool activate/deactivate.

Material Color Changes

Peak color highlighting (EMWaves) and transparency swapping (Gaussian Surface) modify materials directly. To avoid affecting shared material assets, these scripts either use material (creates instance) vs sharedMaterial (modifies asset). Verify this is handled correctly before making material changes — unintended shared-material edits persist in the project files.

Scene Build Settings

All 7 scenes must be registered in File → Build Settings → Scenes In Build, in the correct order. The main menu depends on all scene name strings being correct here. If you add a scene, add it to build settings AND add its button/entry in UIController.
